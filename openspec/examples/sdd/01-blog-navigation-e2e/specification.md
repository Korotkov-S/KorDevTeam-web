# E2E-проверка открытия статей блога

## 1. Контекст

На сайте есть страница `/blog`, где отображаются карточки статей. Данные берутся из API `/api/content/blog?lang=...`, статического индекса `/content/blog.<lang>.json` или fallback-данных компонента `Blog`.

Нужно зафиксировать поведение пользовательского перехода: посетитель открывает список статей, нажимает на карточку и попадает на отдельную страницу статьи.

## 2. Цели

1. Проверить, что страница `/blog` открывается и показывает блок блога.
2. Проверить, что видимые ссылки статей ведут на маршруты `/blog/<slug>`.
3. Проверить, что клик по карточке переводит браузер на новую страницу статьи.
4. Проверить, что на странице статьи есть видимый заголовок и контент.

## 3. Не-цели

- Не проверять весь список статей целиком.
- Не проверять SEO-разметку, sitemap и статическую генерацию страниц.
- Не менять контент статей, изображения и порядок сортировки.
- Не добавлять новые зависимости, если Cypress уже не установлен в окружении.

## 4. Пользовательские сценарии

S1. Посетитель открывает `/blog` и видит список статей.

S2. Посетитель нажимает на видимую карточку статьи.

S3. Браузер переходит на маршрут `/blog/<slug>`, соответствующий ссылке карточки.

S4. Страница статьи отображает `h1` и основной блок `article`.

## 5. Бизнес-правила

- Ссылка карточки статьи должна оставаться внутренним маршрутом вида `/blog/<slug>`.
- Клик по карточке не должен оставлять пользователя на `/blog`.
- Проверка должна быть устойчива к изменению порядка статей.
- Проверка не должна зависеть от конкретного заголовка статьи, если заголовки могут обновляться через контент.

## 6. Критерии приёмки

- E2E-тест открывает `https://kordev.team/blog`.
- Тест находит хотя бы одну видимую ссылку статьи.
- Тест кликает несколько видимых ссылок и проверяет смену `pathname`.
- Для каждой открытой статьи тест проверяет наличие видимого `h1` и `article`.
- OpenSpec-описание поведения остаётся синхронизированным с тестом.
